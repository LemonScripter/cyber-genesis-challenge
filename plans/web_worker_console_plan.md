# [HU] Architektúra Terv: Web Worker JS Console (Valódi Izoláció)

## Cél
A hacker számára egy valódi JavaScript futtatási környezet biztosítása, ahol tetszőleges kódot írhat. A kihívás, hogy ezt úgy kell megoldani, hogy a hacker **ne tudja átírni a szimulátor globális változóit (pl. shieldEnabled, validator)**, de mégis tudjon parancsokat küldeni a virtuális CPU-nak.

## 1. Architektúra: A Causal Bridge (Worker Pattern)
A megoldás egy **Web Worker**. A Web Worker egy teljesen izolált szál a böngészőben, aminek nincs hozzáférése a DOM-hoz vagy a `main.js` globális változóihoz. Csak üzenetekkel (`postMessage`) kommunikálhat a főszállal.

- **Main Thread (A Biztonságos Zóna / BioOS Kernel):** Itt fut a `VirtualCPU`, a `CausalityMonitor` és az `AxiomValidator`. Ide futnak be az üzenetek a Workertől.
- **Worker Thread (A Hacker Homokozó):** Itt fut a hacker által begépelt kód.

## 2. Worker Fájl Létrehozása (`hacker_worker.js`)
Létre kell hozni egy új fájlt, ami a hacker kódját futtatja és kezeli a kommunikációt a BioOS API-val.

## 3. Main Thread Híd Implementálása (`main.js`)
A `main.js`-ben létrehozzuk a Workert, és figyeljük a tőle érkező üzeneteket. Minden ilyen üzenetet **át kell tolni a BioOS Validátoron**, mintha a terminálból jött volna. Itt jön a mágia: A Workerből jövő üzenetnek SOHA nincs érvényes IRQ-ja (mert szoftver generálta a háttérben).

# [EN] Architecture Plan: Web Worker JS Console (True Isolation)

## Objective
Providing a real JavaScript execution environment for the hacker to write arbitrary code. The challenge is to ensure the hacker **cannot overwrite the simulator's global variables (e.g., shieldEnabled, validator)**, while still being able to send commands to the virtual CPU.

## 1. Architecture: The Causal Bridge (Worker Pattern)
The solution is a **Web Worker**. A Web Worker is a completely isolated thread in the browser with no access to the DOM or `main.js` global variables. It can only communicate with the main thread via messages (`postMessage`).

- **Main Thread (Safe Zone / BioOS Kernel):** Runs `VirtualCPU`, `CausalityMonitor`, and `AxiomValidator`. Receives messages from the Worker.
- **Worker Thread (Hacker Sandbox):** Executes code entered by the hacker.

## 2. Worker File Creation (`hacker_worker.js`)
A new file must be created to execute hacker code and handle communication with the BioOS API.

## 3. Main Thread Bridge Implementation (`main.js`)
In `main.js`, we create the Worker and listen for incoming messages. Every message must be **pushed through the BioOS Validator**, just as if it came from the terminal. The magic: Messages from the Worker NEVER have a valid IRQ (since they are generated by software in the background).
